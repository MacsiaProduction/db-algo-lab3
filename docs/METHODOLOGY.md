## 1. Цель и метод

**Лаб задача:** сравнить пять семейств approximate nearest neighbor
(ANN) индексов FAISS на одном дататете (ImageNet-1M, 2048-D L2), снять
trade-off **recall ↔ QPS ↔ память ↔ время сборки**, и выдать
рекомендацию по выбору индекса под разные production-сценарии.

**Метод:** для каждого семейства мы делаем **параметрический свип**
(перебор разумных значений тюнинговых параметров), измеряем 8+ метрик
на каждой точке, и выводим:
- глобальный Парето-фронт по `(recall, qps)`,
- лучшие конфиги на разных порогах Recall@100,
- характеристики масштабирования `100K → 1M`,
- список численно подтверждённых аномалий с объяснениями.

Все измерения — на одной машине, одним FAISS-процессом, в одном
потоке нагрузки (батч из 10 000 query-векторов). Параллельность —
только OpenMP внутри FAISS (8 нитей).

## 2. Датасет

**ImageNet-1M ZJU** — ResNet-50 эмбеддинги ImageNet:

| Split | Размер | Размерность | Тип | Объём |
|---|---:|---:|---|---:|
| Base | 1 281 167 | 2048 | float32 | ≈ 9.4 ГБ |
| Query | 25 000 | 2048 | float32 | ≈ 195 МБ |
| Ground truth | 25 000 × 100 | — | int32 | ≈ 9.6 МБ |

Источник: <http://www.cad.zju.edu.cn/home/dengcai/Data/ANNS/ANNSData.html>.

**Метрика расстояния:** L2 (евклидово). Эмбеддинги не нормализованы
заранее, никакой нашей пред-обработки нет.

**Ground truth:** поставляется заранее посчитанный (через
`IndexFlatL2` на полной базе), но индексирует в IDs > N для любого
N < 1 281 167. Поэтому для свипов с урезанной базой `N_SWEEP` мы
**пересчитываем GT локально** через `IndexFlatL2(base[:N_SWEEP])` и
кешируем в `data/gt_n<N>_k100.npy`.

**Query slice:** все свипы используют первые 10 000 query-векторов
(`LAB_QUERY_N=10000`). 25 000 потребовалось бы для tail-percentiles,
но при наших временных бюджетах это слишком дорого.

## 3. Алгоритмы и их параметры

### 3.1. IVFFlat — `IndexIVFFlat`

**Идея:** база разбивается на `nlist` кластеров k-means'ом. Поиск
сначала находит ближайшие `nprobe` центроидов, затем сканирует
**только их** и возвращает топ-k. Векторы внутри кластеров хранятся
**в исходном виде** (no compression).

**Свипуемые параметры:**
- `nlist ∈ {256, 1024, 4096, 16384}` — число кластеров. Больше →
  тоньше партишн, быстрее поиск при том же `nprobe`, но рост размера
  обучающего набора (нужно `>= 30 · nlist` точек, иначе FAISS warns).
- `nprobe ∈ {1, 4, 16, 64, 256, 1024}` — сколько кластеров посетить.
  Главная knob recall ↔ QPS: `nprobe=1` — быстро и низкий recall,
  `nprobe=nlist` — full scan и recall=1.

**Trade-off:**
- Размер индекса ≈ размер базы (4 байта × dim × N), сжатия нет.
- QPS падает линейно по `nprobe / nlist` (приблизительно).

### 3.2. IVF+PQ — `IndexIVFPQ`

**Идея:** как IVFFlat, но векторы в кластерах **квантуются Product
Quantizer'ом**: разбиваются на `M` суб-векторов размерности `D/M`,
каждый суб-вектор кодируется `nbits` битами (мы используем 8 = 256
codewords). Хранятся **только коды**, ~`M` байт/вектор.

**Свипуемые параметры:**
- `nlist ∈ {1024, 4096}` — то же что у IVFFlat.
- `M ∈ {32, 64, 128}` — число суб-векторов. Делитель 2048: даёт
  64/32/16 D на суб-вектор. **Чем больше M — тем меньше потеря
  точности**, но больше байт/вектор.
- `nbits = 8` (фикс) — 8-битное кодирование, по 256 codewords.
- `nprobe` — как у IVFFlat.

**Trade-off:**
- Размер индекса ≈ `M` байт/вектор + накладные → 60-200 МБ для нашей
  базы (vs 9.4 ГБ у IVFFlat).
- **Потолок recall** определяется потерями квантизации. При 2048 D и
  M=128 (=16 байт/вектор) потолок R@100 ≈ 0.77 — выше не подняться
  даже nprobe=`nlist`.

### 3.3. IVF+SQ — `IndexIVFScalarQuantizer`

**Идея:** как IVFFlat, но каждая компонента вектора квантуется в 8 (SQ8)
или 4 (SQ4) бита независимо. Декомпрессия — на лету при вычислении
дистанции.

**Свипуемые параметры:**
- `nlist ∈ {256, 1024, 4096}`.
- `sq ∈ {SQ4, SQ8}` — точность quantization. SQ8 ~= IVFFlat по recall,
  но 4× меньше по размеру. SQ4 — потеря заметнее, но 8× компрессия.
- `nprobe` — как у IVFFlat.

**Trade-off:**
- Размер: 2 ГБ (SQ8) / 1.2 ГБ (SQ4) при той же N.
- SQ дешевле по компрессии, чем PQ, но SQ8 не теряет recall
  существенно — почти всегда лучший выбор «компрессия + R@100 ≥ 0.95».

### 3.4. HNSW — `IndexHNSWFlat`

**Идея:** мульти-уровневый граф мелкого мира. Каждый вектор — узел,
рёбра идут к `M` ближайшим соседям; верхние уровни разреженные,
нижний — плотный. Поиск — greedy descent от верхнего entry point.

**Свипуемые параметры:**
- `M ∈ {8, 16, 32, 64}` — число рёбер на узел. Больше → плотнее граф
  → выше recall, дольше билд, больше памяти.
- `efConstruction ∈ {40, 100, 200, 400}` — ширина candidate-set при
  построении. Больше → качественнее граф, медленнее билд.
- `efSearch ∈ {10, 20, 40, 80, 160}` — ширина candidate-set при
  запросе. Главная knob recall ↔ QPS.

**Trade-off:**
- Размер индекса = размер базы + ~30 % (граф). У нас ~10 ГБ.
- На дисковых терминах дорого, но recall ≥ 0.99 достигается при QPS ≈ 4 000.

### 3.5. LSH — `IndexLSH`

**Идея:** случайные гиперплоскости (`nbits` штук) разрезают
пространство, каждый вектор кодируется битовой подписью своей
ориентации относительно каждой плоскости. Поиск — Hamming-distance по
сигнатурам.

**Свипуемые параметры:**
- `nbits ∈ {128, 256, 512, 1024, 2048, 4096}` — длина сигнатуры. Чем
  больше — тем точнее, но размер индекса растёт линейно.

**Trade-off:**
- Footprint крошечный (~10–600 МБ).
- При 2048-D случайных гиперплоскостей **недостаточно** — даже
  `nbits=4096` даёт recall@100 ≈ 0.4 на нашем датасете. Включён в лабу как baseline.

## 4. Метрики

Каждая строка в `results/full/<family>.csv` — одна точка свипа.
Колонки:

| Колонка | Что значит | Как меряется |
|---|---|---|
| `algo` | имя семейства | строка |
| `nlist`, `nprobe`, `M`, `nbits`, `sq`, `efConstruction`, `efSearch` | параметры конфигурации | int / строка |
| `build_s` | время train + add (s) | `time.perf_counter()` вокруг train+add |
| `size_mb` | размер сериализованного индекса (МБ) | `faiss.write_index` → `os.path.getsize` |
| `rss_mb` | RSS процесса **после** train + add (МБ) | `psutil.Process().memory_info().rss` |
| `rss_peak_mb` | пик RSS **за время** train + add (МБ) | фоновый sampler 50 мс |
| `rss_delta_mb` | `rss_after - rss_before` для блока train+add (МБ) | разность двух psutil-замеров |
| `faiss_threads` | сколько OpenMP-тредов было активно | `faiss.omp_get_max_threads()` |
| `qps` | пропускная способность (queries/s) | `len(queries) / median_batch_time` |
| `latency_ms` | средняя per-query latency (мс) | `median_batch_time / len(queries) · 1000` |
| `latency_p99_ms` | p99 per-query latency (мс) | **После ре-рана**: p99 по per-chunk таймингам (chunk=50). **До ре-рана** в существующих CSV эта колонка ≈ mean (см. §8). |
| `recall_1`, `recall_10`, `recall_100` | recall@k | `|pred[:k] ∩ gt[:k]| / k`, усреднено по query |
| `n_base` | сколько векторов в базе на момент билда | int |

**О recall@k:** это **не** «доля query, для которых хоть один true сосед
найден», а **«доля совпадений в top-k pred и top-k gt, делённая на k»**.
Это стандарт для ANN. Метрика **не обязана быть монотонной по k**:
например, алгоритм может правильно угадывать top-1, но промазывать по
тонкой структуре top-100 → R@1 > R@10 > R@100.

## 5. Методология замеров

### 5.1. Сборка индекса (build phase)

1. `train_x = utils.load_train_subset(BASE_PATH, 200_000)` — кладёт
   первые 200 000 векторов в memory-resident массив (≈ 1.6 ГБ).
   Используется для k-means обучения у IVF-семейств и для
   `IndexLSH.train`.
2. `with utils.timed(label, sample_rss_peak=True) as tb:` — таймит
   train + add, в фоне 50-мс sampler фиксирует пик RSS.
3. `idx.train(train_x)` — k-means / PQ-обучение / LSH-планы.
4. `utils.stream_add(idx, BASE_PATH, N)` — добавляет `N` векторов
   батчами по `LAB_BATCH_ADD=50000` через `np.memmap` (без
   материализации полной базы в RAM).
5. `size_mb = utils.index_size_mb(idx)` — сериализует индекс во
   временный файл и читает его размер.

### 5.2. Замер QPS / latency

Функция `utils.measure_qps`:
1. `warmup` прогонов `search_fn(queries, k)` — холостые, чтобы
   распрогреть кеши.
2. `repeat` прогонов с тайм-меткой. На каждый прогон батч режется на
   чанки по 50 запросов; каждый чанк таймится отдельно (после
   ре-рана). До ре-рана — `latency_p99_ms` считался по `repeat` (=3)
   замерам всего батча, что давало ≈ mean.
3. `qps = nq / median(batch_times_per_pass)`.
4. `latency_ms = mean(per_chunk_ms across all chunks × repeats)`.
5. `latency_p99_ms = percentile(per_chunk_ms, 99)`.

**Дефолты (`run_all.sh`):** `LAB_QPS_REPEAT=3, LAB_QPS_WARMUP=1`.

### 5.3. Ground truth

- Используется `IndexFlatL2(base[:N])` на тех же векторах, что были
  добавлены в тестируемый индекс. Кешируется в
  `data/gt_n<N>_k100.npy`.
- Recall в CSV — относительно этого локального GT, а не поставляемого
  ZJU GT.

### 5.4. Параллельность

- FAISS использует все доступные OpenMP-нити (`faiss.omp_get_max_threads`).
- Python поток поиска — один.
- Параллельных запросов нет: пользовательский QPS-замер не
  моделирует concurrency, он измеряет batch-throughput одного процесса.

### 5.5. Память: что входит в `rss_peak_mb`

`rss_peak_mb` = `max(rss_before, rss_after, sampled_peak)` для блока
train+add. Это:
- Python-overhead процесса (~80 МБ baseline).
- Тренировочный slice (`train_x`, ≈ 1.6 ГБ).
- Сам индекс на момент пика.
- **Page-cache страниц mmap-нутого `imagenet_base.fvecs`**, который ОС
  считает резидентным у процесса. Для маленьких индексов (IVFPQ, LSH)
  это **доминирующая компонента** (> 90 % пика).

См. график `05_memory_budget.png` для декомпозиции.

## 7. Как читать каждый график

Все графики живут в `docs/img/full/` и генерируются
`scripts/analyze_and_report.py` из CSV в `results/full/`. Один и тот
же скрипт собирает и `DRAFT.md`, и `REPORT.md`.

### 7.1. `05_global_pareto.png` — Cross-algorithm QPS vs Recall@100

Все точки всех семейств в одних осях `(recall@100, QPS)` — QPS в
логарифмической шкале. Цвета — семейства. **Чёрный пунктир** —
глобальный Парето-фронт: точки, которые никто не доминирует
одновременно по обоим осям. **Звёзды** — knee-конфиги каждого
семейства (см. §7.3).

Как читать: если для целевого recall ваша точка ниже пунктира — есть
семейство, которое **строго лучше** на этой recall-зоне. Точки выше
пунктира — best-in-class.

### 7.2. `05_per_family_knees.png` — Per-family Pareto curves

Пять subplots — по одному на семейство. В каждом:
- цветные точки — все конфиги семейства,
- пунктир — Парето-фронт **внутри** семейства,
- золотая ★ — «колено»: точка фронта, ближайшая к идеальному углу
  (recall=1, max QPS) в log-y нормализованной 2D-метрике,
- белые ○ — лучшие конфиги для R≥0.95 и R≥0.99 (если достижимо).

Как читать: даёт картину «как ведёт себя каждое семейство в
изоляции». Полезно для понимания шейпа кривой и того, где находится
«sweet spot».

### 7.3. `05_best_bars.png` — Knee-config comparison

Четыре subplot:
- Build time (s, log) — время на сборку knee-конфига.
- Index size (МБ, log) — размер сериализованного индекса.
- RSS during build (ГБ) — две метрики: «after build» (что осталось в
  RSS после билда) и «peak» (что было в пике). Красный пунктир — 28 ГБ.
- QPS at knee config (log) — пропускная способность того же конфига.

**Что такое knee:** ближайшая к идеальному углу `(Recall@100 = 1,
max QPS)` точка Парето-кривой в нормированном `[0,1]²` с log-y.
Геометрический компромисс по умолчанию: правее тянуть recall дорого
по QPS, левее — recall падает резко.

Как читать: одним взглядом видно, за что мы платим (build_s × index_size
× RSS) ради конкретного `(recall, QPS)`.

### 7.4. `05_memory_budget.png` — Peak RSS decomposed

Стэк-бары: для каждого семейства peak RSS разложен на три полосы:
- серый — **baseline процесса** (Python + ~200 K train slice,
  ≈ 1.7 ГБ),
- синий — **резидентный индекс** = `rss_after − baseline`. Включает
  саму структуру индекса **и** mmap-страницы базы, которые ОС
  держит резидентными у процесса (в serving они тоже занимают RAM,
  поэтому считаем их частью индекса),
- фиолетовый — **transient overhead** = `rss_peak − rss_after`
  (промежуточные буферы и mmap-кеш, освобождаемые после возврата).

Поверх синей полосы подписан `size on disk` — для сравнения с
сериализованным размером (`index.faiss`). Красный пунктир — 28 ГБ
RAM-потолок.

Как читать: у IVFFlat синяя полоса ≈ размер на диске (raw float
векторы); у IVFPQ / LSH синяя полоса много **больше** размера на
диске — это mmap-страницы базы, остающиеся в RAM при serving.

### 7.5. `05_latency_best.png` — Mean + p99 per-query latency

Два бара на семейство в log-шкале: средняя latency и p99 knee-конфига.
p99 считается по per-chunk распределению (50 запросов/чанк,
~200 чанков) в `utils.measure_qps`, поэтому несёт реальный
tail-сигнал, а не «p99 ≈ mean» из per-batch retiming.

### 7.6. `05_scaling.png` — Scaling 100 K → 1 M

Четыре subplot:
- Recall@100 vs N (zoom 0.4–1.02) — насколько устойчив recall с ростом
  N.
- QPS vs N (log) — деградация скорости.
- Build time vs N (log-log) — порядок роста времени сборки.
- Peak RSS during build vs N — рост памяти; пунктир — 28 ГБ.
  Подписана точка, где extrapolation пересекает потолок.

Как читать: видно, какие семейства сохраняют recall при росте базы
(HNSW, IVFFlat, IVFSQ) и какие деградируют (IVFPQ — теряет от шума
при росте плотности).

### 7.7. `05_recall_at_qps.png` — Best Recall@100 at given QPS budget

Для каждого `QPS-floor ∈ [100, 500, 1K, …, 50K]` берётся максимальный
recall@100, который семейство может выдать на конфиге с `QPS ≥ floor`.
Кривые в log-x — recall vs QPS budget.

Как читать: «если у меня бюджет ровно X QPS, то лучший recall, который
я выжму из этого семейства, — Y». Прямой ответ на вопрос «что выбрать
под мой QPS-SLA».

### 7.8. Per-family grids (`05_ivfflat_grid.png`, `05_ivfpq_grid.png`, `05_ivfsq_grid.png`, `05_hnsw_grid.png`, `05_lsh_grid.png`)

По два subplot на семейство: «как растёт recall с параметром поиска»
(слева) и «footprint vs recall» (справа). Цвет — другой параметр
семейства (`nlist`, `M`, `efConstruction` и т.д.).

Что увидеть на каждом:
- **IVFFlat** — recall vs `nprobe` по `nlist`; видно, что больший
  `nlist` выходит на 1.0 раньше.
- **IVFPQ** — recall vs `nprobe` по `(nlist, M)`; плато ≈ 0.79 —
  потолок семейства при 2048-D и M ≤ 128.
- **IVFSQ** — recall vs `nprobe` по `(nlist, SQ4 / SQ8)`; SQ8
  доминирует SQ4 в правом-верхнем.
- **HNSW** — recall vs `efSearch` по `(M, efConstruction)`;
  немонотонность по `efC` при `efSearch ≤ 20` (§8 Tier 1).
- **LSH** — recall vs `nbits`; насыщение ≈ 0.42 даже при 4096 битах.

## 8. Аномалии и data quality

Полный список с доказательствами и root-cause — в `DRAFT.md` §5.
Здесь — категории и принцип.

**Tier 1 (фундаментальные ограничения, не баги):**
- IVFPQ recall ceiling ≈ 0.79 — потолок квантизации при 2048-D и
  16 байт/вектор. Пробить можно только rerank-стадией; см.
  `scripts/tune_ivfpq.py` (`IndexRefineFlat` поверх IVFPQ).
- HNSW recall немонотонен по `efConstruction` при низком `efSearch` —
  свойство greedy-обхода: рыхлый граф сохраняет shortcut-рёбра,
  которые полезны при малом query-бюджете.

**Tier 2 (методологические артефакты sweep ↔ scaling):**
- Build_s расходится между `scaling.csv` и per-family CSV на ~40 %
  для IVF-семейств — `scaling` собран без
  `cp.min_points_per_centroid = 5`, поэтому Lloyd-итераций меньше.
  Recall идентичен (Δ ≤ 0.5 п.п.), различаются только time-based
  метрики.
- `latency_p99_ms` в `scaling.csv` ≈ `latency_ms` — там p99 считается
  по 3 батчевым повторам, в per-family CSV — по per-chunk
  распределению.

**Tier 3 (документированные особенности измерения):**
- Отрицательный `rss_delta_mb` — следствие GC между билдами; не
  ошибка измерения, а свойство per-block разности.
- HNSW peak RSS немонотонен в scaling (между точками N) — следствие
  холодного page-cache после большого IVFFlat-билда. Лечится запуском
  каждого `(family, n)` в отдельном subprocess.
- IVFFlat QPS при `nprobe=1` растёт с `nlist` — cache-эффект: при
  бо́льшем `nlist` партишн помещается в L2-кеш.

## 9. Как пробить потолок IVF+PQ

Стандартный IVF+PQ при 2048-D и `M=128, nbits=8` упирается в
`Recall@100 ≈ 0.79` (см. §8). Этого хватает как кандидатного
генератора, но для self-contained search нужен дополнительный шаг.

`scripts/tune_ivfpq.py` (и его notebook-обёртка `06_pq_tuning.ipynb`)
запускают четыре варианта **в одном файле, без `run_all.sh`**:

1. **IVFPQ** baseline — sanity-check, повторяет ту же конфигурацию,
   что и `02_ivf_benchmark.ipynb`.
2. **OPQ + IVFPQ** — pre-rotation `OPQMatrix(d, M)` оптимизирует
   базис под распределение данных перед PQ. Память та же, +5–8
   процентных пунктов recall.
3. **IVFPQ + Refine** — `IndexRefineFlat` хранит сырые векторы и
   пересортирует top `k × k_factor` PQ-кандидатов по точному L2.
   Память ≈ baseline + `4 × n × dim` байт (≈ 4 ГБ при n=500K).
   `k_factor ∈ {5, 10, 20}` обычно даёт `R@100 ≥ 0.99`.
4. **OPQ + IVFPQ + Refine** — оба вместе; gold standard для
   PQ-based high-recall search.

Результаты пишутся в `results/<mode>/ivf_pq_tuned.csv` со схемой,
совместимой с `ivf_pq.csv` плюс колонками `variant, k_factor, opq,
refine`.

Запуск:

```bash
python3 scripts/tune_ivfpq.py                       # full sweep, n=500K
python3 scripts/tune_ivfpq.py --n 200000            # quick smoke
python3 scripts/tune_ivfpq.py --variants refine     # только Refine
```

Файлы CSV и графики основного отчёта будут перезаписаны командой
`bash run_all.sh && python3 scripts/analyze_and_report.py --run full`.
